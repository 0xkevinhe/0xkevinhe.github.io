共识之迷：一文读懂比特币升级社区进展

2024-12-01
blockchain

Modified: 2024-12-01

本报告内容在 Bitcoin Next, Ep.24 - Bangkok 做了 keynote 演讲，相关 slides 供参考。

前言

关于比特币下一次升级的话题一直不绝于耳，然而到目前（DEC-2024）为止，社区对于要不要升级/升级要解决的问题/要带来的功能等话题并还没有达成一致，基本上是各执一词的情况，像极了某种政治僵局。

在这种僵局下，呈现了很多有趣的现象：

  1. 某些社区成员出于信息不对称或者商业利益，言必称某些 opcode。

  2. 一些项目项目会依赖某些「可能会出现」的 opcode。

考虑到比特币FDV已经快要接近 $2T ，涉及到社会的方方面面，它的升级已经绝对不是某些专业人士通过一次圆桌会议就能决定的。

讨论的前提是信息对等，本文写作的核心目的是为读者提供相关信息，对比特币的升级做相关定义，通过回溯历史来归纳某些规律，进而分析当前的升级提案，最后为读者总结若干 takeaway。意图通过这些信息呈现，为读者对比特币升级话题进行进一步的讨论奠定基础。

同时作者也在文中描述了做调研的过程中引发的对于比特币治理机制的思考。

升级简介：What and Why

什么是比特币升级?

比特币白皮书（https://bitcoin.org/bitcoin.pdf）定义了一个协议，由数以万计遵循比特币协议的节点组成了比特币区块链网络。

协议的实现（通常称为客户端）有很多个版本，根据

https://bitnodes.io/nodes/ 数据源显示，市占比最大的客户端是 Bitcoin Core(https://github.com/bitcoin/bitcoin)，因此 Bitcoin-Core 的代码维护者（下称 Core-Devs 或者 Protocol Developer）在比特币生具备相当重要的影响力。

比特币节点软件有多个模块组成，比特币的相关升级提案由 BIP(Bitcoin Improvement Proposal https://github.com/bitcoin/bips)来定义，人们为 BIP 做了若干分类。

通常情况下，当人们讨论比特币升级，一般是指“共识协议升级”，下文同理，由于共识协议升级需要全网络的大部分的节点形成一致的意见（否则就可能会导致分叉），因此需要特别的慎重。如下图所示，比特币系统中的共识协议相关的模块和 BIP 种共识层有关的提案，值得特别关注。

实际上，根据比特币 github 仓库的统计（https://github.com/bitcoin/bitcoin/pulse/monthly），修改非常活跃，由于大部分的变更都与共识协议无关，因此也没有引起人们的广泛关注。

共识协议升级类型

根据 BIP-123（https://github.com/bitcoin/bips/blob/master/bip-0123.mediawiki）的定义，共识协议的升级主要分为软分叉（soft fork）与硬分叉（hard fork）。

特性 Soft Fork Hard Fork
是否兼容旧节点 是 否
是否需要全网更新 不一定 是
是否有链分裂风险 低 高
示例 SegWit/Taproot Bitcoin XT/Bitcoin Cash

  • 另外，还有一个不是那么直觉的方式来诠释和对比，也很有趣：

    Soft fork：增加/加强规则（简单想象一下，增加了一个新功能，例如支持 taproot 地址）

    Hard fork：移除/放松规则（简单想象一下，移除了一个限制，例如移除了区块奖励的限制）

BIP和软分叉的流程

前两次成功的共识协议升级（Taproot/SegWit），均使用软分叉的方式，在不出现巨大的社区分裂的情况下，本文集中讨论soft fork（软分叉），即是兼容老版本软件的情况下进行升级。

通常一个软分叉提案会聚合多个 BIP，例如 taproot 就包含了 3 个 BIP：

  1. Schnorr Signature(BIP-340 https://github.com/bitcoin/bips/blob/master/bip-0340.mediawiki)
  2. Taproot(BIP-341 https://github.com/bitcoin/bips/blob/master/bip-0341.mediawiki)
  3. Tapscript(BIP-342 https://github.com/bitcoin/bips/blob/master/bip-0342.mediawiki)

回顾一下 Taproot 的升级的时间表：

典型软分叉的阶段里程碑包括：

  1. 对应的 BIP 被提出，实现方案通过 review
  2. Bitcoin-Core 代码维护者发起升级 github pull request
  3. Bitcoin-Core 代码维护者审核和合并 github pull request，决定激活方式
  4. 矿工链上投票方式来决定激活区块高度
  5. 新版本 Bitcoin-Core 代码发布
  6. 区块高度到达约定高度，完成升级

为什么要升级

从前文关于升级的描述，读者可以发现，这是一个相当冗长的过程。有读者可能就会问了，不做升级性行不行？作者做了一下更新与不更新的利弊分析。

Consensus Change Pros Cons
Change Technological AdvancementEnhanced SecurityExpanded Use Cases Risk of ForkingIncreased Complexity
Unchanged Stability and TrustAvoids Split RisksMinimizes Attack Surface Technological StagnationLacks Flexibility for New Demands

总的来讲，从「可持续性」与「适应性」等角度来看，根据需要来持续升级，是一种更加可取的方式。

升级深入

升级的利益相关方

比特币历史上的香港共识（签署于2016 年 2 月的比特币圆桌活动，参考 https://medium.com/@bitcoinroundtable/bitcoin-roundtable-consensus-266d475a61ff）主要参与方是：

  1. Core-Devs
  2. 矿池
  3. 用户和生态开发者（主要是交易所/芯片厂商等）

随着比特币的采用率的快速提升，比特币升级的利益相关方也从最早简单的三权分立逐步发展演变，进入了列王纷争局面，参考https://github.com/bitcoin-cap/bcap 报告。

这里面几个角色值得重点介绍一下：

  1. Economic Nodes：主要是指主流的 CEX 交易所/支付机构/托管服务商等，他们的在soft fork种的态度决定哪种是合法的比特币，会对采用率有重要影响。
  2. Investors：在比特币策略（EFT/机构储备/国家储备等）在全球盛行的大背景下，投资者这个角色本身变得更加复杂。
  3. User&Ecosystem Developer：在 Taproot 升级之后，比特币生态蓬勃发展，出现了 Ordinals 等资产协议，也涌现了一大批原生应用/扩容协议。

对于这些角色，有一些有趣的结论：

  1. 不同的利益相关方在不同的阶段发挥不同作用：例如 Ecosystem Developer 对于提案有比较大的积极性，Core-Devs行使对于BIP 审核权限，矿池和经济节点对于激活有比较大的影响力
  2. 不同的 Ecosystem Developer 倾向提出和支持与自己商业利益相关的提案：例如 Validity Rollup 的支持者会倾向于支持 OP_CAT
  3. Protocol Developer（等同Core-Devs）：这个角色当前会更多倾向于否决某些升级提议，而不是积极推动某些提议。

升级的历史和总结

根公开资料，从比特币网络启动以来，发生过多次 soft fork 升级。

数据来源：

https://blog.bitmex.com/a-complete-history-of-bitcoins-consensus-forks-2022-update/

https://www.drivechain.info/media/slides/mit-2023.pdf

从上图中可以总结出一些有趣的结论：

  1. 比特币的协议出现了某种僵化，随着时间推移， softfork 的频次降低
  2. 升级的共识达成需要的时间越来越长

Soft Fork 关注方面

分析过往的 soft fork 包含的 BIP，可以总结出有如下的关注方面：

关注方面 案例
扩展性（scalability） SegWit/Schnorr
隐私（privacy） Taproot/MAST
可编程性（programability) CLTV/Tapscript
安全性（security） Disable Opcodes

何为好的升级提案

根据前面的各方面列举的事实和分析，我们尝试定义一个好的升级提案：

  1. 坚持比特币作为支付系统的核心定位：比特币是有独特定位的
  2. 应用潜力/带来风险之间取得优雅的平衡：使得大部分人都喜欢，没有人强烈反对
  3. 合适的升级规模：不能太简单（不值得大动干戈），也不能太复杂（推动很困难）
  4. 合理的时机：需要有强烈的需求，解决特定的问题。例如在 SegWit 升级阶段，扩容是强需求

升级展望

提案归类

作者收集了大部分活跃的提案，将它们归类在四个关注象限中，便于读者进行可视化理解。

社区呼声

从上图可以看到，社区对于升级要解决的问题有一定的共识，即围绕支付系统所需要的功能扩展，可以归类为如下2 大类：

  1. 可编程性：使得 UTXO 的更强的编程能力，例如 covenant/vault/交易自省/条件支付/script 增强等（例如 CTV）
  2. 扩展性：对于 L2 的扩展，整体方案又分为链上验证和链下验证两大类，都有一些积极主推的提案(例如 CAT)

共识之谜

可以认为比特币社区对于下一次升级陷入共识的迷宫，原因如下：

  1. 大到不敢动：接近 $2T FDV 的软件系统，相当部分的利益相关方倾向于保持稳定，没有哪一方愿意承担事故责任
  2. 利益相关方高度分化： 不同利益相关的诉求不一样，在不同的阶段能发挥的作用不一样；政府也成了利益相关者
  3. 治理机制缺乏：比特币作为最早的区块链，缺乏链上治理的机制；社区对于软分叉激活方式也没法达成共识
  4. Core-Devs角色变化：倾向于做否决而不是推动提议
  5. 缺乏紧迫性：区块链基础设施发展日趋完善，对于比特币的升级没有非常强的需求

总结&Takeaway

本文通过介绍比特币升级的基础概念，对历史升级进行了深入分析，最后展望了下一次升级的活跃提案，归纳出当前存在的共识之迷的原因。

经过回顾和展望，相信读者已经对于当前升级所处的状态有一定了解，最后总结若干 takeaway。

  1. 升级是必不可少的，软分叉更可取
  2. 利益相关高度分化，社区倾向于保守
  3. 必须在坚持比特币核心价值定位的前提下谈升级
  4. 扩展性只是其中一个升级的关注方面
  5. 需要一个更好的时机，好的升级提案会快速获得共识
  6. 社区需要探索更好的治理机制
